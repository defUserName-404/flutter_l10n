# Flutter L10n

Android Studio / IntelliJ plugin for extracting Dart string literals into ARB files, replacing source strings with localization getters, and running `flutter gen-l10n`.

## Highlights

- PSI-based extraction (no regex scanning)
- Review dialog to edit value/key before writing
- ARB upsert semantics (manual value edits are persisted)
- Riverpod-first replacement (`ref.l10n.key`), with `context.l10n.key` fallback
- Static-context safety (`/* TODO ... static context */` instead of invalid code)
- Multi-locale dialog with optional Ollama batch translation
- Auto-import management and support file generation

## Installation

### Build and install from disk

```bash
./gradlew buildPlugin
```

Install `build/distributions/flutter-l10n-1.0.0.zip` from:

- Android Studio / IntelliJ -> Settings -> Plugins -> gear icon -> Install Plugin from Disk

### Development sandbox

```bash
./gradlew runIde
```

## Quick Start After Install

1. Add `l10n.yaml` to your Flutter project root.
2. Ensure your ARB directory and template file exist (for example `assets/l10n/app_en.arb`).
3. Ensure your localization output files are generated by Flutter (`flutter gen-l10n`).
4. Provide Riverpod localization wiring (`l10n_providers.dart`) if you want `ref.l10n` calls to resolve.
5. Open a Dart file and run **Extract Strings to L10n** (`Ctrl+Shift+L` / `Cmd+Shift+L`).

## Usage Flow

1. Trigger action from editor context menu, Tools menu, or shortcut.
2. Plugin extracts candidate strings from PSI.
3. In the review dialog:
   - keep/uncheck rows
   - edit `Value`
   - edit `Key`
4. If multiple locales are available, optional translation dialog opens.
5. Plugin updates ARB files, rewrites source usages, ensures imports, and runs `flutter gen-l10n`.

## Replacement Strategy

The replacement is resolved per string location:

- Riverpod scope (`WidgetRef`, `Ref`, `Notifier`) -> `ref.l10n.<key>`
- BuildContext in scope -> `context.l10n.<key>`
- Static context -> TODO comment (safe skip)
- Unknown scope fallback -> `ref.l10n.<key>`

## ARB Upsert Rules

For each selected row (`key`, edited `value`, optional locale translations):

- Template locale (from `template-arb-file`):
  - always overwrite key value with edited value
  - always ensure metadata block (`@key`) exists
- Non-template locale:
  - if translation provided for locale: overwrite with provided translation
  - if key exists and no new translation: keep existing translation
  - if key is new and no translation: write `TODO(key): <template-value>`

This guarantees manual edits from the review dialog are reflected in ARB files.

## Configuration

### `l10n.yaml`

Supported keys loaded by the plugin:

```yaml
arb-dir: assets/l10n
template-arb-file: app_en.arb
output-localization-file: app_localizations.dart
output-class: AppLocalizations
output-dir: lib/core/config/l10n
preferred-supported-locales:
  - en
  - fr
synthetic-locale: false
nullable-getter: false
```

### Translation (Ollama)

Current defaults are code-based (`TranslationConfig`):

- host: `http://localhost:11434`
- endpoint: `/api/generate`
- model: `phi3`
- timeout: `120000 ms`

If Ollama is unavailable, translation requests return empty/null values and manual entry still works.

## Architecture (DI + SOLID)

Package structure under `com.defusername.flutter_l10n`:

- `action` - IDE action entrypoint
- `core` - composition root and config loader
- `extraction` - orchestration contracts + implementations
- `arb` - locale discovery and ARB upsert writer
- `replacement` - scope-aware replacement strategy
- `detection` - PSI scope detectors
- `imports` - import and support-file management
- `gen` - `flutter gen-l10n` runner
- `translation` - translation client and parser
- `ui` - dialogs

The action itself is intentionally thin. Orchestration is handled by `ExtractionCoordinator`, which depends on interfaces (`StringExtractor`, `SelectionReview`, `ArbRepository`, etc.) and is wired by `ServiceGraph`.

## Troubleshooting

- Action not visible:
  - open a file in editor (presentation requires active editor)
- `ref.l10n` unresolved:
  - ensure your project has Riverpod localization provider/extension setup
- `flutter gen-l10n` warning:
  - ensure Flutter is on `PATH` or set `FLUTTER_ROOT`
- Translation not filling automatically:
  - ensure Ollama is running and model is available
